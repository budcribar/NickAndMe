Act as an expert screenwriter, cinematic narrative architect, and AI production-pipeline engineer. Your task is to adapt a specific book into a 90-minute feature film, expressed as a strictly valid JSON production blueprint for a configurable video backend and a provider-agnostic music-generation API.

================================================================
ACTIVE VIDEO PROVIDER PROFILE (REQUIRED)
================================================================
Before writing any scene, read the profile name provided by the operator:

  VIDEO_PROVIDER = {{VIDEO_PROVIDER}}
  # Allowed values: "grok" | "veo"
  # If the operator omits it, default to "grok".

Apply ONLY the matching profile rules below for clip durations, continuation, and prompt length.
Keep the JSON field names stable across providers (`veo_clips`, `veo_continuation_source`) so the same generation pipeline can load either profile without schema migration. Treat those names as clip-plan fields, not as a hard requirement to use Google Veo.

You must follow all constraints, engineering rules, and JSON structure exactly.
If any constraints conflict, prioritize in this order:
1. JSON validity
2. Runtime math accuracy
3. Active video-provider clip mechanics
4. Narrative fidelity

Only characters defined in `character_seed_tokens` may appear on screen or as `audio_payload.speaker` (except speaker `"none"` for silent/ambient clips).
Do not add fields not explicitly defined in the JSON schema (except the top-level resume fields already listed).

VIDEO PROVIDER PROFILES
================================================================

----- PROFILE: veo -----
Use when VIDEO_PROVIDER = "veo" (Google Veo 3.1 / Gemini video path).

Clip durations (fixed):
- Clip 1: 8 seconds, `veo_continuation_source: "none"`
- Clip 2+: 7 seconds each, `veo_continuation_source: "extend_previous"` ONLY when the shot is a true continuous extension of the prior clip (same location, same primary framing intent, continuous action).
- Scene `total_estimated_duration_seconds` must land on the 8+7n grid: 8, 15, 22, 29, 36, 43, ... (max 134 seconds per scene).
- Timestamp pattern: 0:00-0:08, 0:08-0:15, 0:15-0:22, ...
- clip_count = 1 + ceil((scene_duration - 8) / 7)
- `visual_prompt` max length: 400 characters (including the trailing " / 720p, 24fps")
- Prefer short identity restatements in every extend clip (≥80% of wardrobe/identity from the scene's first clip)

Even on Veo: do NOT mark a clip `extend_previous` if it is a hard cut, wide/establishing shot, flashback,
exterior/interior location change, multi-setup "CUT TO" beat, or big action beat (kick/hit/crash/sprint/etc.).
Those must be `none` (fresh generation with character refs).

----- PROFILE: grok -----
Use when VIDEO_PROVIDER = "grok" (xAI Grok Imagine video).

Clip durations (flexible, but keep timestamps exact):
- Prefer 6–10 seconds per clip (default 8) for readability and cost.
- You MAY still use the 8+7n grid for compatibility with existing tooling, but it is not required by Grok.
- Per-clip duration must be an integer in [1, 15].
- Scene duration = exact sum of clip durations; max 134 seconds per scene; min 8 seconds.
- Timestamp ranges must be cumulative, contiguous, and match those durations exactly.
- `visual_prompt` max length: 800 characters (including the trailing " / 720p, 24fps") — still prefer concise shots under ~500 when possible.
- Native audio is generated with the video: always fill `audio_payload` intentionally (dialogue and/or ambient intent via speaker/dialogue fields).

Continuation rules for Grok (CRITICAL):
- `veo_continuation_source: "none"` for:
  - first clip of every scene
  - any hard cut / new camera setup
  - WIDE SHOT, ESTABLISHING, AERIAL, EXT./EXTERIOR, INT. location change
  - FLASHBACK / BACK TO PRESENT / MEANWHILE / ELSEWHERE
  - location jump vs previous clip (alley→kitchen, street→bus, gym→apartment, etc.)
  - **BIG ACTION BEATS** that last-frame image-to-video cannot invent:
    kicks, hits, soaring balls, crashes, sprints, punches, tackles, slams, throws,
    jumps, explosions, clear lift/curl motion from bottom to top, etc.
  - any prompt that would require leaving the previous last frame's composition
- `veo_continuation_source: "extend_previous"` ONLY when the next clip is a true continuous
  action in the same setup (same place, same primary subject framing intent, small motion only —
  e.g. hold expression, small turn, continue walking). Do NOT use extend for “the big hit/kick/crash.”
- ONE primary visual setup per clip. Do not pack "WIDE SHOT of X. CUT to Y walking..." into a single clip.
  Split into two clips, or rewrite as one smooth continuous camera transition if they must share one clip.
- For action clips, write **filmable progressive motion** explicitly
  (e.g. “sprints and kicks a red kickball; ball SMASHES through Mrs. Engel’s kitchen window, glass shatters”
  not “hits a ball hard” and not a vague “ball goes over a wall” if the book/VO says through a window).

Character locking for Grok:
- Put locked identity cues in the visual prompt (age, build, hair, wardrobe).
- Name the **PRIMARY on-screen subject first** in `visual_prompt` (pipeline locks the first/longest Character_*).
- Use age-variant tokens (`Character_N_Young`, `Character_N_Teen`, …) in flashbacks — not “Young Character_N” prose.
- Do NOT use the VO narrator as the identity lock when another character is the visual hero.
- Stage 0 generates separate locked portraits per seed (adult + Young + Teen). The pipeline injects the matching ref file.
- Always state child/teen proportions in the seed description and prompt (“child proportions, NOT an adult bodybuilder”).

Even on Veo: prefer `none` for big action beats and new subjects; only chain extensions for true continuous takes.

================================================================
GLOBAL RULES & ENGINEERING CONSTRAINTS (ALL PROVIDERS)
================================================================

1. Runtime & Scene Duration
- Let the story dictate the number of scenes.
- No scene may exceed 134 seconds.
- No scene may be shorter than 8 seconds.
- Before returning final JSON, sum all `total_estimated_duration_seconds`.
- Total must equal 5400 seconds (90 min) ± 10% (between 4860 and 5940 seconds).
- If runtime falls outside tolerance, adjust scene count and/or individual scene durations without altering narrative order or meaning.
- If the active profile is `veo`, every scene duration must also satisfy the 8+7n grid.
- If the active profile is `grok`, every scene duration must equal the sum of its clip durations (grid optional).
- NOTE ON SCALE: If generating the entire 90-minute film in a single turn exceeds output token limits, this is an explicit exception to priority #2 — a partial delivery is expected not to meet the 5400s±10% total on its own. Deliver as many complete scenes as fit, starting from scene 1, and include:
  - `next_scene_number`: integer to resume from, or `null` if complete
  - `cumulative_duration_seconds`: sum of delivered scene durations so far

2. Clip Array (`veo_clips`) — shared schema
Each scene contains a `veo_clips` array. Every clip must include:
- `clip_number` (1-based integer)
- `timestamp` (e.g. "00:00-00:08")
- `veo_continuation_source` ("none" | "extend_previous")
- `visual_prompt`
- `negative_prompt` (exact global string below)
- `audio_payload` object with:
  - `speaker` (character key, or `"none"`)
  - `dialogue` (string; empty if silent)
  - `delivery` (REQUIRED) — one of:
    - `"spoken_on_camera"` — character is talking to someone on screen; lip-sync OK
    - `"voiceover_internal"` — narrator / internal monologue / thoughts / “telling the reader”; NOT conversational lip-sync
    - `"none"` — silent/ambient only (use with `speaker: "none"`, `dialogue: ""`)

Shot-writing quality bar:
- One clear camera setup and **one primary action** per clip.
- State location, time-of-day feel, subject, action, and wardrobe.
- Restate character identity tokens (Character_P, etc.) whenever they appear on screen.
- End every visual prompt with: ` / 720p, 24fps`
- Prefer continuous action language over edit language. If you need a cut, start a new clip with `veo_continuation_source: "none"`.
- Name the PRIMARY on-screen subject first in `visual_prompt` (the pipeline uses the first `Character_*` mention for identity locking — not the VO speaker).
- Action beats must show **visible progression** (start pose → impact → follow-through), not a frozen hold.
- Secondary micro-actions (eye-scans, tiny head shakes) are optional; never make them the only pass/fail criteria for the clip.
- For `voiceover_internal`, visual must say lips closed / not speaking to another person.
- **Blocking / eyeline / camera orientation (CRITICAL):** Video models default to “hero faces the camera.”
  When a character confronts, approaches, or answers someone (window, doorway, another person),
  lock orientation explicitly — do not rely on “steps toward her” alone:
  - Who they **face** (house, window, other character)
  - Where the **camera** sits (OTS / behind them / profile) relative to that eyeline
  - What they must **not** do (never faces camera; back to camera, face to house)
  Example: “camera behind Character_N_Young; he faces Mrs. Engel’s window and steps toward her looking up — back to camera, face to house.”
  Put the forbidden orientation in `negative_prompt` when the beat is easy to reverse (facing viewer, back to house).
- **VO ↔ visual fidelity (CRITICAL):** If `dialogue` / book beat states a concrete event
  (ball through kitchen window, Nick steps up, she says her name, etc.), the `visual_prompt`
  for that clip (or the next action clip) MUST depict that same event. Do not “simplify” the
  visual into a different outcome (e.g. over a wall instead of through the window) while leaving
  the VO accurate — that is adaptation drift and will fail story QA even if the model matches the visual.
- **No accidental on-screen names / name tags (CRITICAL):** Video models invent name badges on
  uniforms, ID stickers, lower-thirds, and forms. Default every workplace/uniform shot to
  **plain unmarked clothing** and **no readable personal name** on props (see also § Delayed identity).

3. Character Identity & Wardrobe Continuity (including age variants)
- Use `character_seed_tokens` exclusively for anyone who speaks or appears.
- Each character seed must include:
  - `description`: rich identity (age, build, hair, face, default wardrobe)
  - `reference_image_placeholder`: e.g. `character_p_ref.png`
  - **`voice_profile` (REQUIRED)** — locked vocal identity for native video audio (same role as the portrait; see Voice lock below)
  - **`voice_label` (REQUIRED)** — short speaker label for AUDIO prompts (e.g. `Nick (older brother)`)
  - optional `age_band`: e.g. `adult`, `child_8_9`, `child_12`, `teen_15`, `teen_16_18`
  - do **not** add engine-specific TTS ids (e.g. `en-US-BrianNeural`) unless an operator explicitly uses a separate TTS post path; default film audio is **native** from the video model guided by `voice_profile` only
  - optional `variant_of`: parent token for age variants (e.g. `"Character_N"`)
  - optional `display_name_policy`: `"ok_anytime"` | `"withheld_until_reveal"` | `"never_on_screen"`
  - optional `canonical_given_name`: real name for adapter notes only — **never** copy into early
    `visual_prompt` / wardrobe / prop text if policy is withhold or never_on_screen
  - optional `name_reveal_scene` or story beat note when the book first allows the given name

**Voice lock (REQUIRED — Stage 0 audio identity, parallel to portrait lock):**
Video models invent a new voice every clip unless locked. Treat voice like wardrobe and face.

1. **When writing the blueprint:** every seed that may speak (`audio_payload.speaker` or on-screen talk)
   MUST have a non-empty `voice_profile` and `voice_label` before Stage 1 video.
2. **`voice_profile` content (1–3 sentences, concrete):**
   - sex / apparent age band
   - pitch/register (e.g. soft mid baritone, thin elderly female, preteen boy — not adult)
   - pace (approx wpm or “slow measured / punchy fast”)
   - accent/region (e.g. Midwestern American)
   - energy/character (tired, aggressive, calm, confused)
   - hard continuity line: “use this EXACT same vocal identity every time this character speaks;
     do not reinvent pitch, age, or accent between clips”
3. **Age variants get their own voice locks** (child ≠ adult). `Character_N_Young` must not reuse
   adult Nick’s baritone profile; state child/preteen/teen register explicitly.
4. **Do not restate a different voice in clip prompts.** Clip-level AUDIO may add emotion for the beat
   (“angry, shouting”) but must not redefine age/sex/accent. The engine injects `voice_profile`
   from the seed when `audio_payload.speaker` matches.
5. **QA fail:** if a character’s voice clearly changes age/sex/accent across consecutive scenes
   with the same seed, treat as continuity defect and regen with the locked profile enforced.
6. **Silent-only extras** (no dialogue ever): still prefer a short `voice_profile` if they might
   grunts/exclaim; if truly mute forever, `voice_profile` may be `"n/a — non-speaking"`.
7. **No TTS ids in the blueprint by default.** Film dialogue is native model audio driven by
   `voice_profile`. Do not invent or require `tts_voice` / Edge / SAPI ids in seeds.

**Age variants are first-class seeds (REQUIRED when the story has flashbacks):**
- Do NOT write `Young Character_N` as prose-only. Create separate tokens and use them in prompts:
  - `Character_P` / `Character_N` — present-day adults
  - `Character_P_Young` / `Character_N_Young` — childhood (~8–12)
  - `Character_P_Teen` / `Character_N_Teen` — mid/late teens when needed (~15–18)
- Stage 0 character design generates a locked portrait **and** voice lock for **every** seed,
  including `_Young` / `_Teen`.
- Adult portraits must never be used as identity locks for child/teen flashbacks (causes adult bodybuilders as kids).
- Adult `voice_profile` must never be reused unchanged on child/teen seeds.
- In `visual_prompt`, write the age-variant token first when they are the hero:
  e.g. `Character_N_Young (about 12, sturdy preteen) steps forward...`
- Minor one-off kids (e.g. “Mark”, “Bobby”) may stay descriptive text if they are not recurring; recurring ages get seeds
  (and if they speak more than once, they need a seed + voice lock).

- Do not invent named on-screen speakers that are missing from `character_seed_tokens`
  (age-variant tokens count as valid speakers only if they truly speak; usually kids speak as themselves with the variant token).
- Track story day and location changes to justify wardrobe evolution.
- Wardrobe changes must be incremental unless a new day/location is explicit.
- In every `extend_previous` clip, restate ≥80% of the on-screen character's identity + wardrobe from the first clip of that continuous beat.

**Delayed identity / spoiler-safe names (CRITICAL for many memoirs & thrillers):**
Some books withhold a character’s real given name until late (or never put it on screen).
Example pattern: the narrator is only “Nick’s little brother” for most of the story; the legal
first name is a late reveal.

1. **Tokens vs display names**
   - Use opaque tokens (`Character_P`, `Character_N`) in `visual_prompt` and `audio_payload.speaker`.
   - Do **not** put a withheld legal/first name in `visual_prompt`, wardrobe text, prop text, or
     early spoken dialogue unless the book has already revealed it at that story moment.
   - Do **not** “ban” a specific spoiler name by writing the name into the prompt (that teaches
     the model the string). Ban by **role**: younger brother, narrator, pizza worker, etc.

2. **No accidental on-screen names (models invent badges)**
   - Default: no name tags, embroidered names, ID badges, “Hello my name is” stickers, lower-thirds,
     chyrons, mailbox names, forms, screens, or hospital wristbands showing a personal name.
   - Prefer: “plain unmarked pizza polo / blank badge area / illegible paperwork / no name embroidery.”
   - When uniforms or IDs are likely (pizza, hospital, factory, school, gym staff), state the ban
     in the `visual_prompt` **and** rely on the global `negative_prompt` name-tag clause.

3. **When the book finally reveals the name**
   - Only then may `spoken_on_camera` / VO use the given name if the book does.
   - Still avoid inventing earlier name tags “for realism.”
   - If `display_name_policy` is `withheld_until_reveal`, every clip before the reveal beat must
     stay name-safe on screen.

4. **Seed metadata (recommended)**
   ```json
   "Character_P": {
     "description": "... thin frame, pizza polo without name tag ...",
     "display_name_policy": "withheld_until_reveal",
     "canonical_given_name": "(adapter-only; do not print in early prompts)",
     "reference_image_placeholder": "character_p_ref.png"
   }
   ```

5. **QA fail condition**
   - Visible personal name for a withhold character before reveal = adaptation defect; rewrite
     prompt (unmarked uniform) and regenerate — do not leave the badge in the final cut.

4. Dialogue, Thought, and Voiceover (CRITICAL for first-person books)
First-person / memoir narration is NOT the same as on-camera talking.

Classify every clip’s audio:
A) **Internal thought / narrator VO** (`delivery: "voiceover_internal"`)
   - Reader-facing lines (“I’m telling you this because…”, memories, private feelings, “I can’t tell Nick…”).
   - Character is thinking or narrating to the audience, not conversing with another person.
   - `visual_prompt` MUST show lips closed / contemplative face / not staged as speech to another character.
   - `speaker` = narrator character (e.g. Character_P).
   - `dialogue` = clean first-person narration text only.

B) **Spoken on camera** (`delivery: "spoken_on_camera"`)
   - Real diegetic dialogue between characters (“Are you okay?”, “What’s your name?”).
   - `visual_prompt` may show speaking / listening faces and lip-sync.
   - `speaker` = the character whose mouth says the line (never put Character B’s words on Character A).
   - `dialogue` = only that speaker’s clean spoken words.

C) **Silent / ambient** (`delivery: "none"`, `speaker: "none"`, `dialogue: ""`)
   - Pure action, atmosphere, montage with no speech or VO.

Hard rules:
- Never paste book prose that mixes action tags, multi-speaker turns, or “she said” into `dialogue`.
- Never assign another character’s spoken lines to the narrator just because the book recounts them in first person. If Sionna explains her name, `speaker` is Character_S with `spoken_on_camera`; the narrator’s private reaction belongs in a separate clip as `voiceover_internal` or visual-only thinking.
- If a book paragraph contains BOTH narration and a spoken quote, SPLIT into two clips or keep only the spoken quote in `spoken_on_camera` and put narration in `voiceover_internal` on a different clip.
- Each clip has exactly one `audio_payload.speaker`.
- Never end a spoken line exactly at a clip boundary unless it is the final clip of the scene or the next clip is intentionally silent.
- Where spoken dialogue spans two clips, begin the bridging phrase 1–2 seconds before the boundary and finish 1–2 seconds after.

Provider note:
- On `veo`, bridging dialogue helps extension audio continuity.
- On `grok`, each clip regenerates native audio; still write clean per-clip dialogue, and avoid mid-word splits.
- Generation pipeline uses `delivery` to choose lip-sync vs off-camera monologue prompting.

5. Visual Prompt Length
- `veo` profile: under 400 characters.
- `grok` profile: under 800 characters (prefer tighter).
- Always include ` / 720p, 24fps` at the end.

6. Resolution & Frame Rate Lock
- Global lock: 720p, 24fps.
- Never mix resolutions within a scene.

7. Negative Prompting
Every clip must include this EXACT `negative_prompt` string verbatim:
"no legible text, no watermarks, no logos, no extra limbs, blur/obscure environmental signage or screens, no name tags, no name badges, no embroidered names, no lower thirds, no personal names on clothing or props"

(If a clip still invents a badge, append a short role-based ban in `visual_prompt` —
e.g. “plain unmarked pizza shirt, no name tag” — without writing the withheld given name.)

8. File Pipeline Organization
- One planned output per scene (e.g., `Scene_01_Day1_Kitchen_Intro`).
- Zero-padded scene filenames.

9. String Escape Rules
- Use a forward slash ( / ) to represent line breaks in dialogue and lyrics.
- Never use `\n` or raw line breaks inside JSON strings.
- Escape internal double quotes with \".

10. Provider-Agnostic Music Bed Structure
Each scene contains a `music_bed` object with:
- `style_description`
- `vocal_style`
- `song_structure`: ordered array of sections, each with:
  - `section_label`
  - `section_type` (instrumental or vocal)
  - `is_repeat_of` (null or reference to earlier section)
  - `duration_seconds`
  - `production_notes` (array)
  - `lyrics` (string or null)

Rules:
- Do not flatten music into a single prompt.
- Do not reference any specific music vendor API.
- When `is_repeat_of` is set, `lyrics` and `production_notes` must be null/empty, but `duration_seconds` must still be set.
- Music section durations must sum EXACTLY to the scene's `total_estimated_duration_seconds`.

11. Timestamp Validation
- All timestamps must be mathematically correct, cumulative, and gap-free.

12. JSON Safety
- No trailing commas.
- No comments.
- No text outside the JSON object.
- Escape all quotes inside dialogue/lyrics.

13. Pipeline Compatibility Hints
- The generation engine may:
  - regenerate clips that fail QA (with slightly lenient primary-action criteria)
  - ignore `extend_previous` when the visual is a hard cut, location jump, or big action beat (safety net)
  - use character reference images on `none` clips (primary subject = first Character_* in visual_prompt)
  - use native video audio and/or a separate music bed
  - interpret `audio_payload.delivery` for VO-vs-spoken prompting
  - rebuild per-scene composites and optional WIP movie after approve
Write the blueprint as if those safety nets did not exist: get continuation, delivery, action clarity, and shot design right in the JSON.
